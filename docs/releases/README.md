# Release Notes

Structured GitHub Release bodies live in this directory as one file per tag.

- File name format: `v<version>.md`
- Example: `v1.0.0.md`
- Scope: write user-facing highlights, authentication notes, quality notes, and explicit scope boundaries

Release publishing behavior:

- If `docs/releases/<tag>.md` exists, the release workflow publishes that file as the release body
- If the file does not exist, the workflow falls back to GitHub's autogenerated release notes
- Before verification and zip creation, the release workflow runs `pnpm preflight:release`. This fails closed if `WXT_GITHUB_APP_CLIENT_ID`, `WXT_GITHUB_APP_SLUG`, or `WXT_GITHUB_APP_NAME` is missing or empty, preventing a package with disabled private-repository sign-in from being uploaded.

Suggested structure:

1. Release summary
2. `Highlights`
3. `Authentication`
4. `Quality & Packaging`
5. `Scope Notes`
6. `Full Changelog`

## v1.8.0

Minor release for **GitHub Pulls Show Reviewers** focused on runtime contract
hardening, stronger TypeScript safety, clearer platform policy, and broader
auth lifecycle coverage. The extension remains intentionally narrow: requested
reviewers, requested teams, and completed review state are still the only
pull-list metadata surfaced by the product.

### Highlights

- Runtime messages are now backed by zod schemas, giving background, content,
  and options entrypoints explicit validation boundaries before data crosses
  extension contexts.
- TypeScript now uses stronger optional-property checks across the repository,
  tightening internal contracts without changing the reviewer UI.
- Browser support and language policy are documented: Chrome remains the
  supported browser target, and the current MVP remains English-only while
  future localization costs are tracked explicitly.

### Authentication

- No permission, token-scope, storage-schema, or host-permission changes.
- Public repositories still work without signing in when GitHub's
  unauthenticated REST API allows it.
- Private repositories continue to use the maintainer-owned GitHub App account
  flow with `Pull requests: Read` access only.
- Packaged-extension coverage now exercises the private-repository auth
  lifecycle, including account sign-in, token refresh, and disconnect behavior.

### Quality & Packaging

- Unit coverage now has enforced thresholds so reviewer, routing, settings,
  and diagnostics regressions are harder to miss.
- End-to-end coverage now includes private auth lifecycle behavior in the
  packaged MV3 extension.
- Release packaging behavior is unchanged: GitHub Actions still runs the
  release preflight, `pnpm verify:release`, and Chrome zip creation before
  attaching the package to the GitHub Release.

### Scope Notes

- Reviewer visibility on GitHub pull request list pages remains the only
  product goal.
- Checks, mergeability, labels, assignees, and general PR dashboard features
  remain out of scope.

## v1.4.4

Patch release for **GitHub Pulls Show Reviewers** focused on clearer access
guidance when reviewer fetches fail. The reviewer feature scope stays
unchanged: requested reviewers, requested teams, and completed review state
remain the only surfaced PR-list metadata.

### Highlights

- Access banners now use an explicit `BannerKind` taxonomy with deterministic
  priority ordering, so the highest-signal private-repository or rate-limit
  guidance wins when multiple rows fail for different reasons.
- Unauthenticated 401, 403, and 404 responses without a rate-limit signal are
  routed to sign-in guidance instead of being grouped with transient or
  unattributed failures.
- Reviewer-fetch envelopes now carry a `rateLimited` signal through the
  runtime boundary, allowing content-script guidance to distinguish GitHub API
  limits from regular access failures.
- Rate-limit detection now matches GitHub's documented primary and secondary
  rate-limit messages without treating unrelated text such as "no rate limit
  applied" as a limit signal.

### Authentication

- No permission, token-scope, storage-schema, or host-permission changes.
- Public repositories still work without signing in when GitHub's
  unauthenticated REST API allows it.
- Private repositories still use the maintainer-owned GitHub App with
  `Pull requests: Read` only.

### Quality & Packaging

- Added regression coverage for banner priority, failure classification, and
  rate-limit signal propagation across the reviewer-fetch runtime path.
- Added API regression coverage for primary and secondary GitHub rate-limit
  message detection plus a false-positive guard.
- Updated implementation and manual Chrome testing notes to document the
  banner-kind behavior.

### Scope Notes

- Reviewer visibility on GitHub pull request list pages remains the only
  product goal.
- Checks, mergeability, labels, assignees, and general PR dashboard features
  remain out of scope.

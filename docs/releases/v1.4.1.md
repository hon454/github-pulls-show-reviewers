## v1.4.1

Patch release for **GitHub Pulls Show Reviewers** focused on access guidance and release confidence. The reviewer feature scope stays unchanged: requested reviewers, requested teams, and completed review state remain the only surfaced PR-list metadata.

### Highlights

- Access banners now name the current repository when the GitHub App installation does not cover it, making the next action clearer for organization-owned private repositories.
- The uncovered-repository banner link now says `Configure access`, matching the GitHub App installation action more directly than a generic sign-in prompt.

### Authentication

- No permission, token-scope, storage-schema, or host-permission changes.
- Public repositories still work without signing in when GitHub's unauthenticated REST API allows it.
- Private repositories still use the maintainer-owned GitHub App with `Pull requests: Read` only.

### Quality & Packaging

- PR CI now runs the same release gate signals as `pnpm verify:release`: lint, typecheck, unit tests, and Playwright e2e coverage. The jobs are split so fast-path checks and e2e run in parallel.
- README release workflow notes were synchronized with the expanded CI coverage.

### Scope Notes

- Reviewer visibility on GitHub pull request list pages remains the only product goal.
- Checks, mergeability, labels, assignees, and general PR dashboard features remain out of scope.

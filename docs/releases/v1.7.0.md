## v1.7.0

Minor release for **GitHub Pulls Show Reviewers** focused on keeping reviewer
chips fresh on GitHub pull request list pages. The reviewer feature scope stays
unchanged: requested reviewers, requested teams, and completed review state
remain the only surfaced PR-list metadata.

### Highlights

- Reviewer cache entries now track freshness so cached reviewer chips can render
  immediately while stale rows revalidate in the background.
- Re-requested reviewers and requested teams can update on pull-list pages
  without requiring a full browser reload.
- Same-repository GitHub SPA render and navigation events now trigger targeted
  reviewer revalidation when row state may have changed.
- Existing-row mutation handling uses lightweight row fingerprints so unrelated
  DOM changes avoid unnecessary reviewer API requests.

### Authentication

- No permission, token-scope, storage-schema, or host-permission changes.
- Public repositories still work without signing in when GitHub's
  unauthenticated REST API allows it.
- Private repositories continue to route through the maintainer-owned GitHub
  App account flow, and reviewer revalidation uses the same background-worker
  token handling as existing reviewer fetches.

### Quality & Packaging

- Added regression coverage for freshness helpers, stale rerendering,
  same-repository render revalidation, and mutation fingerprint behavior.
- Refreshed user-facing README guidance so the setup path and reviewer display
  behavior are easier to scan before installation.
- Chrome Web Store submission materials now point at the v1.7.0 package name.

### Scope Notes

- Reviewer visibility on GitHub pull request list pages remains the only
  product goal.
- Checks, mergeability, labels, assignees, and general PR dashboard features
  remain out of scope.

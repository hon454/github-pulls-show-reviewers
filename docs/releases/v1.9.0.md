## v1.9.0

Minor release for **GitHub Pulls Show Reviewers** focused on reviewer-list
performance and GitHub DOM resilience. This release keeps the product scope
unchanged: requested reviewers, requested teams, and completed review state
remain the only pull-list metadata surfaced by the extension.

### Highlights

- Reviewer metadata batching now follows bounded, validated GitHub REST
  pagination so searched, filtered, and older paginated pull request lists can
  reuse page-level metadata instead of falling back to per-row requests.
- Visible pull request numbers are sent through the metadata batch runtime
  contract, allowing the background fetch path to stop once the visible rows are
  covered.
- Reviewer rows can still mount when GitHub renders a pull request list variant
  without the usual metadata container.

### Authentication

- No permission, token-scope, storage-schema, or host-permission changes.
- Public repositories still work without signing in when GitHub's
  unauthenticated REST API allows it.
- Private repositories continue to use the maintainer-owned GitHub App account
  flow with `Pull requests: Read` access only.
- Page-level auth, access, not-found, and rate-limit failures are cached after
  fallback-account retry is exhausted so the same page avoids repeating row
  fallback requests that would hit the same failure.

### Quality & Packaging

- Adds fixture-backed coverage for GitHub pull request list markup without the
  expected metadata container.
- Adds regression coverage for metadata batch target pull numbers, validated
  pagination, fallback suppression, and fallback-account cache race handling.
- GitHub CI passed `lint-and-test` and `e2e` for PR #107 before this release.
- Release packaging behavior is unchanged: GitHub Actions still runs the
  release preflight, `pnpm verify:release`, and Chrome zip creation before
  attaching the package to the GitHub Release.

### Scope Notes

- Reviewer visibility on GitHub pull request list pages remains the only
  product goal.
- Checks, mergeability, labels, assignees, and general PR dashboard features
  remain out of scope.

## v1.10.0

Minor release for **GitHub Pulls Show Reviewers** focused on GitHub pull-list
DOM resilience and packaged-extension regression coverage. This release keeps
the product scope unchanged: requested reviewers, requested teams, and completed
review state remain the only pull-list metadata surfaced by the extension.

### Highlights

- Pull request number detection now prefers GitHub's primary title link and
  falls back to navigation pull links when the title link loses the expected
  primary-link class.
- Fallback reviewer mount placement now follows that same link precedence, so
  reviewer UI stays anchored beside the pull request title instead of an earlier
  navigation link.
- Same-page row fingerprinting now uses the centralized pull-link selector path,
  keeping mutation handling aligned with reviewer mount and pull-number parsing.

### Authentication

- No permission, token-scope, storage-schema, or host-permission changes.
- Public repositories still work without signing in when GitHub's
  unauthenticated REST API allows it.
- Private repositories continue to use the maintainer-owned GitHub App account
  flow with `Pull requests: Read` access only.
- Reviewer fetching and fallback-account behavior are unchanged.

### Quality & Packaging

- Added fixture-backed coverage for pull-list markup where the title link keeps
  navigation behavior without GitHub's primary-link class.
- Expanded packaged-extension e2e coverage around reviewer rendering, options
  boot behavior, and access-banner paths.
- GitHub CI passed `lint-and-test` and `e2e` for PR #115 before this release.
- Release packaging behavior is unchanged: GitHub Actions still runs the
  release preflight, `pnpm verify:release`, and Chrome zip creation before
  attaching the package to the GitHub Release.

### Scope Notes

- Reviewer visibility on GitHub pull request list pages remains the only
  product goal.
- Checks, mergeability, labels, assignees, and general PR dashboard features
  remain out of scope.
